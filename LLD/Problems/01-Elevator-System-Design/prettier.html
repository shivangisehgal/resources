The Golden Rule of LLD Interviews
Before you write a single class
START THINKING FROM THE CLIENT'S PERSPECTIVE.
Don't jump into classes. First ask: what does the user actually do? Trace a user story end-to-end. Every class you design should exist because a user action demanded it — not because it seemed like a good idea.

The user flow for an elevator is simple and concrete. Use it to anchor yourself:

User at floor 3 → presses UP/DOWN button (outside elevator) → External Request →
System decides which elevator opens → Elevator Selection →
User enters, presses floor 7 → Internal Request →
Elevator moves to floor 7 → Done
Two types of requests exist. This is the core split that drives the whole design.
• External Request: From outside the elevator — user presses UP or DOWN on a floor. Needs: (direction, src floor)
• Internal Request: From inside the elevator — user presses a destination floor button. Needs: (dest floor, elevator ID)
I
Requirements Gathering
Ask these questions before designing

In an interview, always clarify requirements first. Here is what to ask/confirm for an elevator system:

  • R1 Multiple elevators — How many? Are they identical or different types?
  • R2 Who are the customers? — Is this a building system, a simulation, or an API?
  • R3 Elevator system scope — Is it just the controller logic, or also hardware simulation?
  • R4 Kinds of elevators — Passenger, Service, Private (VIP)? These may differ in behavior.
  • R5 Capacity — Weight or person limit?
  • R6 Range of floors — Basement? Penthouse? Zone restrictions?
  • R7 Algorithm of movement — Which floor scheduling algorithm? LOOK, SCAN, FCFS, Shortest Seek Time? Should it be changeable?
  • R8 Status — Does the system need to track IDLE, MOVING, NOT_WORKING states?
Instructor note from your class: Things like fan, light, buttons inside the elevator are "irrelevant in this scope." Know what is in-scope and out-of-scope. For this problem: focus on the movement logic and request routing, not internal hardware components.
II
Start from main()
Your entry point reveals your top-level design

A powerful LLD trick: write what your main() would look like. This forces you to think about the public API of your system before designing internals.

public static void main(String[] args) { // 1. Get the singleton instance of ElevatorSystem ElevatorSystem system = ElevatorSystem.getElevatorSystem(); // 2. Initialize: how many floors, how many elevators system.initialiseElevatorSystem(10, 3); // 3. Set the algorithm (elevator selection strategy) system.setElevatorSelectionStrategy(new OddEvenElevatorSelStrategy()); // 4. External request: someone on floor 3 wants to go UP system.sendExternalRequest(ELEVATOR_DIRECTION.UP, 3); // 5. Internal request: once inside elevator #2, go to floor 7 system.sendInternalRequest(7, 2); }
What main() tells you: You now know ElevatorSystem is a Singleton. You know it has an initialise method, a setAlgo method, and two send-request methods. You've designed the top-level API without touching any internal class yet.

V
Design Patterns Used
Why each pattern fits naturally
Singleton
ElevatorSystem and ElevatorMgr are both singletons. A building has exactly one elevator system. You don't want two instances accidentally creating duplicate state. Use a private constructor + static getInstance() method.
Strategy
Used in TWO places:
① ElevatorSelectionStrategy — which elevator answers an external call? Options: OddEvenStrategy, ZoneStrategy.
② ElevatorControlStrategy — how does a selected elevator decide its next stop? Options: LOOK algo, SCAN algo, FCFS, Shortest Seek Time. Both can be swapped at runtime — "you can change it once a year, month, whenever."
Observer
Sensor observes elevators. When a floor change is detected via detectFloor(), it notifies registered displays. The notifyElevators(floorNum) method pushes the update. Displays are subscribers.
State
ElevatorState captures the current condition of the elevator (direction, status, floor). The controller reads this state when deciding movement. Encapsulating state separately makes it easy to extend without changing the elevator itself.
VI
The Two Strategy Layers Explained
The most important design decision in this system

Layer 1: Which elevator should respond? (ElevatorSelectionStrategy)

When someone presses UP/DOWN on floor 3, which elevator should open? The ExternalRequestsProcessor holds a strategy for this. Examples:

OddEvenElevatorSelStrategy
selectElevator(ExternalRequest) : int
If src floor is odd → pick odd-numbered elevator; even → even elevator.
ZoneElevatorSelStrategy
selectElevator(ExternalRequest) : int
Floors 1–5 → elevator zone A, Floors 6–10 → elevator zone B.

Layer 2: What floor should it stop at next? (ElevatorControlStrategy)

Once an elevator is moving, it needs to decide: what's the next floor to stop at? The ElevatorController holds this strategy. Examples:

LookAlgoElevatorControlStrategy
determineNextStop(floorNum) : int
Goes in one direction until no more requests, then reverses (like a look-ahead scan).
ScanAlgorithmECS
determineNextStop(floorNum) : int
Goes end-to-end like a disk scan. Services requests in order of direction.
ShortestSeekTimeECS
determineNextStop(floorNum) : int
Goes to whichever pending floor is closest to current floor.
FirstComeFirstServeECS
determineNextStop(floorNum) : int
Processes requests in the order they arrived. Simple queue logic.

End-to-End Request Flow
Trace a call from button press to elevator arrival

External Request (button pressed outside elevator)

User Action
User on floor 3 presses UP. Creates: ExternalRequest(UP, srcFloor=3)
ElevatorSystem.sendExternalRequest()
Delegates to ExternalRequestsProcessor.processExternalRequest(req)
Elevator Selection
ERP calls elevatorSelectionStrategy.selectElevator(req) → returns elevator ID (e.g., 2)
Get Elevator
ERP asks ElevatorMgr.getElevator(2) → gets Elevator #2 object
Move
Calls elevator.controller.moveElevatorToFloor(3). Controller uses its ElevatorControlStrategy to decide movement steps.

Internal Request (button pressed inside elevator)

User Action
Inside elevator #2, user presses floor 7. Creates: InternalRequest(destFloor=7, elevatorId=2)
ElevatorSystem.sendInternalRequest()
Delegates to InternalRequestsProcessor.processInternalRequest(req)
Get Elevator + Move
IRP gets elevator #2 from ElevatorMgr, then calls controller.moveElevatorToFloor(7)

Interview Execution Tips
How to present this in a real interview
  • 🎯
    Always start with requirements. Spend 2–3 minutes asking clarifying questions. Interviewers reward this. It shows you don't just blindly code.
  • 🧭
    Draw the user flow first. Before any class, say: "A user presses UP on floor 3. Let me trace what happens." This grounds your design in reality.
  • 📋
    Write a fake main() early. This is a power move. It forces you to define your public API before internals, and shows you think top-down.
  • 🔀
    Name your design patterns out loud. "I'm using Strategy here because the algorithm needs to be swappable at runtime." Showing pattern awareness is a huge green flag.
  • ⚖️
    Know when NOT to add a pattern. "Factory could work here, but for this scope it's overkill." Being deliberate about simplicity shows seniority.
  • 🏷️
    Use ENUMs for all fixed states. Direction (UP/DOWN), Status (IDLE, MOVING, NOT_WORKING). Never use magic strings or ints.
  • 📦
    Separate concerns clearly. Selection strategy (which elevator) vs Control strategy (which floor next) are different responsibilities. Don't merge them.
  • 🔑
    Singleton justification. "ElevatorSystem is a singleton because there is only one building-level elevator system. Two instances would cause inconsistent state." Always explain why, not just what.
